# Забезпечення якості коду та Рефакторинг

## 1. Виявлення запахів коду (Code Smells) та їх усунення

Під час аналізу початкового коду (зокрема у FastAPI та Django сервісах) були виявлені наступні запахи коду:

1. **God Class / Fat Controller (main.py / views.py):** Вся бізнес-логіка (створення замовлення, перевірка продуктів через HTTP, логіка кошика, збереження в БД) містилася безпосередньо у функціях роутерів (у `main.py`). Це робило код важким для читання, підтримки та неможливим для Unit-тестування без підняття БД.
2. **Дублювання коду (Duplicated Code):** Логіка перевірки наявності товару або зміни статусу замовлення повторювалася в кількох місцях (наприклад, при підтвердженні оплати та при доставці).
3. **Жорстка зв'язність (Tight Coupling):** Функції роутера напряму працювали з сесією БД (`db.add()`, `db.commit()`), не дозволяючи легко замінити ORM чи базу даних.
4. **Magic Numbers / Magic Strings:** Використання хардкоджених URL-адрес (`http://product_service:8000`) та статусів у вигляді звичайних строк, замість `Enum`.

## 2. Проведений Рефакторинг (Чистий Код та SOLID)

Для усунення цих проблем у сервісах (`order_service`, `payment_service`, `rating_service`) було застосовано архітектурні зміни на основі SOLID:

- **Single Responsibility Principle (SRP):** Виділено три окремі шари:
  - **Routers (`main.py`):** Відповідають лише за прийом HTTP запиту, виклик сервісу та повернення HTTP відповіді.
  - **Service Layer (`service.py`):** Концентрує в собі всю бізнес-логіку (взаємодія зі Stripe, перевірка лімітів відгуків, логіка корзини). Не знає про HTTP чи SQL.
  - **Repository Layer (`repository.py`):** Відповідає виключно за операції з базою даних (ORM SQLAlchemy). Це дозволяє приховувати SQL-запити від бізнес-логіки.
- **Open/Closed Principle (OCP):** Створено базові класи для репозиторіїв та DTO (Pydantic), що дозволяє розширювати функціонал без зміни існуючого коду.
- **Dependency Inversion Principle (DIP):** За допомогою механізму Dependency Injection у FastAPI (`Depends()`), сервіси отримують свої залежності (Repository) ззовні. Це кардинально спростило написання тестів та зменшило зв'язність (coupling).

## 3. Модульне та інтеграційне тестування

Після рефакторингу логіка стала ізольованою, що дозволило покрити її тестами за допомогою **PyTest**:

- **Unit тестування (Модульне):** Написані тести для `OrderService` (у `tests/test_service.py`), де замість реальної бази даних використовується `MagicMock` для `OrderRepository`. Це гарантує тестування виключно бізнес-логіки (наприклад, перевірка правильності розрахунку ціни, обробка помилок при відсутності товару) без залежності від БД та інших мікросервісів.
- **Integration тестування:** Реалізовані інтеграційні тести для API ендпоінтів (`tests/test_api.py`) за допомогою `TestClient` FastAPI. Для них підіймається тестова база даних SQLite (`in-memory`), і перевіряється весь ланцюг: від HTTP запиту до збереження в БД. Забезпечено **>50% покриття** основної логіки.

## 4. Якість коду та Лінтери

В проєкті використовуються інструменти статичного аналізу:
- **Flake8:** Для перевірки на відповідність стандарту PEP8 (запобігає синтаксичним та стилістичним помилкам).
- **Mypy:** Для перевірки статичної типізації (Type Hints) у Python.
- **ESLint / Prettier:** Для підтримки чистого та уніфікованого коду на frontend (React/TypeScript).
- **Pytest-Cov:** Інструмент генерації звіту про покриття коду тестами (Coverage Report).

Впроваджено CI/CD конвеєр (через GitHub Actions), який автоматично запускає ці перевірки при кожному пуші в репозиторій. Це є невід'ємною частиною забезпечення якості у командній розробці.
